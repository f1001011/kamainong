产品信息说明：

产品体系规则：
- VIP等级仅为产品分类名称，用户本身无VIP等级概念

- 赠送产品：管理员后台可向指定用户赠送产品，赠送的产品与购买的产品独立展示，收益发放规则相同；管理员可随时撤回赠送产品，撤回后产品在用户前端静默消失，历史已发放收益不回滚，未来收益暂停发放
- 赠送产品附加规则：
  - 赠送的产品让该用户算"有效会员"（满足有效会员判定条件）
  - 赠送的产品不触发上级的团队三级返现（仅用户自己花钱购买才触发返现）
  - 赠送的产品给用户抽奖次数（与购买产品规则一致，VIP等级数=抽奖次数）
- 每个产品有独立图片，后台上传管理，前端产品列表和详情页展示
- 产品列表展示顺序：默认VIP0在最前、VIP9在最后，后台支持拖拽配置排序

提现完整流程（6个阶段 + 状态机）：

状态流转图：
```
用户提交提现
    │
    ▼
┌──────────────┐     免审核通过      ┌──────────┐
│ PENDING_REVIEW │ ──────────────→ │ APPROVED │
│  （待审核）    │                   │ （已通过） │
└──────┬───────┘                   └─────┬────┘
       │                                 │
  管理员审核                        自动/手动代付
       │                           submitTransfer()
       ├─── 通过 ──→ APPROVED ──→ 调用支付通道
       │                              │
       └─── 拒绝 ──→ REJECTED        ├── 接口返回失败 ──→ PAYOUT_FAILED
                    （退回余额）       ├── 接口异常       ──→ PAYOUT_FAILED
                                      └── 接口成功       ──→ 等待回调
                                                              │
                                           回调成功 ──→ COMPLETED（扣冻结余额）
                                           回调失败 ──→ PAYOUT_FAILED（保持冻结）
                                                              │
                                                    管理员处理 ┤
                                                              ├─ 重试代付 → APPROVED → 重走代付流程
                                                              └─ 驳回退款 → FAILED（退回余额）
```

阶段1：用户前端申请提现
- 入口：用户提交提现请求
- 检查提现门槛（必须已购买VIP1-VIP9产品）
- 检查时间窗口（10:00-17:00）
- 检查今日次数（每日1次）
- 检查金额范围（最低55 TJS）
- 检查余额是否充足
- 获取银行卡信息，生成快照
- 检查免审核条件
- 事务处理：创建订单 + 冻结余额 + 记录资金流水
- 如果满足免审核条件 → 状态直接为APPROVED，异步调用代付
- 如果不满足免审核条件 → 状态为PENDING_REVIEW，等待管理员审核
- 关键逻辑：提现金额从可用余额（availableBalance）原子转移到冻结余额（frozenBalance）
- 免审核条件（5项全满足）：总开关开启 + 金额≤阈值 + 在免审核时间窗口内 + 今日次数未超限 + 不在新用户冷却期

阶段2：管理员后台审核
- 审核通过：检查订单状态必须是PENDING_REVIEW → 更新状态为APPROVED + 记录审核人/时间 → 异步调用代付（不阻塞返回）
- 审核拒绝：检查订单状态必须是PENDING_REVIEW → 事务内订单状态改为REJECTED + 冻结余额退回可用余额 + 创建WITHDRAW_REFUND资金流水 + 发送站内通知

阶段3：提交支付通道代付
- 获取订单，状态必须是APPROVED
- 获取代付通道（指定通道或自动选择可用通道）
- 读取银行卡快照
- 确定通道订单号（首次=原始订单号，重试=生成新号，避免通道重复订单号）
- 构建代付参数（金额、银行编码、账号、姓名、手机号、回调URL等）
- 记录代付尝试记录（payoutAttempts）
- 调用通道创建代付订单
- 通道接口返回成功 → 等待异步回调（仅表示"申请受理"，最终结果以回调为准）
- 通道接口返回失败或网络异常 → 标记为PAYOUT_FAILED
- 重试场景：重新生成通道订单号，避免支付通道拒绝重复订单号
- payoutAttempts数组记录每次代付尝试（通道、订单号、状态、时间）

阶段4：支付通道异步回调
- 接收通道回调 → 验签 → 分布式锁防并发
- 幂等处理：已完成/已失败/已拒绝的订单直接返回
- 状态必须是APPROVED
- 通道校验：防止旧通道延迟回调干扰重试后的订单
- 回调成功 → 事务处理：状态改为COMPLETED + 扣减冻结余额 + 记录流水 + 发送通知
- 回调失败/冲正 → 状态改为PAYOUT_FAILED + 余额保持冻结 + 不通知用户
- 关键设计：代付失败不自动退款，只标记PAYOUT_FAILED，余额保持冻结，等管理员处理
- 用户端展示：PAYOUT_FAILED状态在前端统一显示为APPROVED（处理中），对用户透明

阶段5：管理员处理代付失败
- PAYOUT_FAILED后管理员有两个选择：
- 选项A — 重试代付（可换通道）：检查状态为PAYOUT_FAILED或APPROVED → 检查重试次数≤5次 → 验证目标通道可用 → 更新订单（状态→APPROVED，channelId→新通道，重置回调数据，retryCount+1）→ 异步调用代付
- 选项B — 驳回退款：检查状态必须是PAYOUT_FAILED → 事务内订单状态改为FAILED（终态）+ 冻结余额退回可用余额 + 创建WITHDRAW_REFUND资金流水 + 发送通知"提现失败，已退回余额"

阶段6：定时轮询补偿机制
- 扫描条件：APPROVED + 已有通道ID + 更新超5分钟 + 10天内的订单
- 每次最多处理30单
- 逐个调用通道查询代付订单状态
- 上游成功 → 补偿入账（扣冻结余额 + 状态改COMPLETED）
- 上游失败 → 状态改PAYOUT_FAILED（保持冻结，等管理员处理）
- 上游处理中 → 跳过
- 管理员也可手动触发查询上游，同样会触发自动补偿

提现状态总结对照表：
| 阶段 | 触发方 | 状态变化 | 余额操作 |
|------|--------|----------|----------|
| 用户提现 | 用户 | → PENDING_REVIEW / APPROVED | 可用余额 → 冻结余额 |
| 审核通过 | 管理员 | PENDING_REVIEW → APPROVED | 无 |
| 审核拒绝 | 管理员 | PENDING_REVIEW → REJECTED | 冻结余额 → 可用余额 |
| 提交代付 | 系统 | APPROVED（不变） | 无 |
| 代付接口失败 | 系统 | APPROVED → PAYOUT_FAILED | 保持冻结 |
| 回调成功 | 支付通道 | APPROVED → COMPLETED | 扣减冻结余额 |
| 回调失败 | 支付通道 | APPROVED → PAYOUT_FAILED | 保持冻结 |
| 重试代付 | 管理员 | PAYOUT_FAILED → APPROVED → 重走代付 | 保持冻结 |
| 驳回退款 | 管理员 | PAYOUT_FAILED → FAILED | 冻结余额 → 可用余额 |
| 轮询补偿 | 定时任务 | APPROVED → COMPLETED / PAYOUT_FAILED | 成功扣冻结 |

支付通道技术架构（参考已有平台）：

整体设计思想：
- 采用策略模式 + 工厂模式 + 数据库配置化
- 所有通道实现同一个 PaymentChannel 接口（10个方法）
- 通道凭证（商户号、密钥、网关地址）存数据库，运行时加载
- 通道管理器（工厂）根据 code 字段自动创建对应通道实例
- 业务层只与接口交互，完全不关心具体通道实现细节

目录结构：
```
apps/api/src/lib/payment/
├── types.ts              # 统一接口 + 所有类型定义
├── base-channel.ts       # 基类（MD5、排序签名、HTTP、日期格式化）
├── channel-manager.ts    # 通道管理器（工厂 + 缓存 + 数据库加载）
├── index.ts              # 统一导出
└── [xxx]pay.channel.ts   # 各通道实现文件
```

核心接口（PaymentChannel）— 每个通道必须实现10个方法：
```
interface PaymentChannel {
  readonly code: string;   // 通道代码，如 'ALLAPAY'
  readonly name: string;   // 通道显示名

  // ====== 代收（充值）5个方法 ======
  createCollectionOrder(params)    // 创建充值订单 → 返回支付页面URL
  verifyCollectionCallback(payload) // 验证充值回调签名 → 返回标准化回调数据
  getCollectionCallbackSuccessResponse() // 返回通道要求的成功响应格式
  getCollectionCallbackFailResponse()    // 返回通道要求的失败响应格式
  queryCollectionOrder(orderNo)    // 主动查询充值订单状态

  // ====== 代付（提现）5个方法 ======
  createTransferOrder(params)      // 创建代付订单 → 提交提现请求到通道
  verifyTransferCallback(payload)  // 验证代付回调签名 → 返回标准化回调数据
  getTransferCallbackSuccessResponse()  // 返回通道要求的成功响应格式
  getTransferCallbackFailResponse()     // 返回通道要求的失败响应格式
  queryTransferOrder(orderNo)      // 主动查询代付订单状态
}
```

基类（BaseChannel）提供的工具方法：
| 方法 | 作用 |
|------|------|
| md5(str) | MD5哈希 |
| sortAndJoin(params) | 参数按ASCII排序拼接成 k=v&k=v |
| toFormUrlEncoded(params) | 转form-urlencoded格式 |
| formatDateTime(date?) | 格式化日期 YYYY-MM-DD HH:mm:ss |
| parseGatewayDateTime(dateStr) | 解析网关时间（时区转换） |
| formatAmount(amount) | 金额保留2位小数 |
| httpPost(url, data) | 发送form POST请求并解析JSON |

通道管理器（PaymentChannelManager）：
```
数据库 payment_channels 表
        │
        ▼ (启动时加载，Redis缓存5分钟)
┌─────────────────────────┐
│  PaymentChannelManager  │  (单例)
│                         │
│  createChannel(config)  │ ← switch(config.code) 创建对应实例
│  getChannel(code)       │ ← 业务层调用入口
│  refresh()              │ ← 后台修改通道后刷新缓存
│  verifyCallbackIp()     │ ← 回调IP白名单校验
└─────────────────────────┘
```
- 密钥安全：数据库中的 paySecretKey / transferSecretKey 使用 AES-256-CBC 加密存储，管理器加载时自动解密（兼容未加密的旧数据）

| 字段 | 类型 | 说明 |

| payFeeMode | VARCHAR(20) | 充值手续费模式：FIXED / PERCENT / MIXED |
| payFeeFixed | DECIMAL | 充值固定手续费金额 |
| payFeePercent | DECIMAL | 充值百分比费率（如2.5表示2.5%） |
| transferFeeMode | VARCHAR(20) | 代付手续费模式：FIXED / PERCENT / MIXED |
| transferFeeFixed | DECIMAL | 代付固定手续费金额 |
| transferFeePercent | DECIMAL | 代付百分比费率（如2.5表示2.5%） |

回调路由：
- 统一回调入口，动态路由自动匹配通道
- POST /api/callback/{channel}/{type}
  - /api/callback/xxxpay/recharge → 充值回调
  - /api/callback/xxxpay/withdraw → 提现回调
- 路由只做3件事：解析请求体 → 分发给对应service → 返回通道要求的响应格式
- 新增通道只需在白名单数组中加一项

接入新通道标准步骤（5步）：
1. 创建通道文件 — xxxpay.channel.ts，继承BaseChannel，实现PaymentChannel接口的10个方法
2. 注册到管理器 — channel-manager.ts 的 createChannel 中加一个case
3. 导出模块 — index.ts 加一行export
4. 回调路由加白名单 — route.ts 中白名单数组加上新通道code
5. 数据库插入通道配置 — INSERT payment_channels 记录，清Redis缓存或重启服务生效

调用链路总览：
```
用户充值:  前端 → /api/recharge/create → rechargeService → channelManager.getChannel(code)
                                                           → channel.createCollectionOrder()
                                                           → 返回 payUrl（跳转第三方收银台）

充值回调:  支付平台 → /api/callback/{channel}/recharge → rechargeService.handleCallback()
                                                        → channel.verifyCollectionCallback()
                                                        → 验签通过 → 入账

用户提现:  前端 → /api/withdraw/create → withdrawService → 冻结余额 → (免审核/审核通过)
                                                           → submitTransfer()
                                                           → channel.createTransferOrder()

提现回调:  支付平台 → /api/callback/{channel}/withdraw → withdrawService.handleTransferCallback()
                                                        → channel.verifyTransferCallback()
                                                        → 验签通过 → 完成/标记失败

状态轮询:  定时任务 → channel.queryCollectionOrder() / queryTransferOrder() → 补偿入账或标记失败
```

团队三级返现：
- 返现比例：L1（直推）27% | L2（二级）2% | L3（三级）1%

渠道管理：

用途说明：
- 管理员在平台冷启动阶段创建渠道推广链接，用于投放到不同推广渠道（如Telegram群、Facebook广告、地推团队等）
- 渠道链接格式与用户前端邀请链接完全一致，注册用户无法区分来源
- 每个渠道链接拥有独立的邀请码，所有通过该链接注册的用户归属该渠道

- 渠道链接仅用于统计计数和数据追踪，不影响正常的团队返现逻辑（如渠道链接邀请了A，A邀请了B，B购买产品时A正常获得L1返现）

九、常见问题

1、伊泰普水电站建设的重要性是什么？
伊泰普水电站能够为巴西和巴拉圭提供大量清洁、可再生能源，对两国电力供应具有重要意义。

该项目不仅促进了两国经济的可持续发展，还有效减少了碳排放。

由于水电站位于两国边境，因此由巴西和巴拉圭政府共同管理，这也形成了 Itaipu Binacional 这一双边合作机构。

2、伊泰普水电站在哪里？
伊泰普水电站位于巴西和巴拉圭边界的巴拉那河上。

具体位置为：
巴西巴拉那州的福斯-杜伊瓜苏（Foz do Iguaçu）市
以及巴拉圭的埃斯特城（Ciudad del Este）。

3、如何在 Itaipu Binacional 工作？
Itaipu Binacional 采用公开、公平的招聘制度，类似于公共考试制度，以确保所有人都能公平竞争岗位。

这一制度体现了公司在机会公平方面的承诺。

4、为什么 Itaipu 被称为"双边公司"？
因为 Itaipu 是由巴西和巴拉圭两国根据条约共同建立和管理的企业，目的是共同开发巴拉那河的水电资源。

5、伊泰普水电站位于哪个城市？
伊泰普水电站位于两座城市之间：

巴西：Foz do Iguaçu
巴拉圭：Ciudad del Este

两座城市分别位于两国边境。
