# 渠道链接管理功能 - 完整方案

> 最后更新：2026-02-08 v3（最终版，已验证全部代码引用）

---

## 一、功能概述

在管理后台新增「渠道链接」功能模块。管理员可为指定用户创建渠道，系统自动基于该用户的邀请码生成推广链接。通过此功能，管理员可追踪每个渠道的推广效果。

**核心设计理念：** 渠道链接 = 某个用户邀请码的管理别名。不改变现有注册/邀请/返佣逻辑，仅在管理层增加一个追踪维度。

**链接格式：** 与用户邀请链接完全一致 `https://{site_domain}/register?ref={user.inviteCode}`，域名从 `GlobalConfig` 的 `site_domain` 键获取。

---

## 二、名词定义

| 名词 | 定义 | 计算方式 |
|------|------|----------|
| 渠道 | 管理员创建的推广追踪单元，绑定一个用户的邀请码 | — |
| 一级下线 | 通过渠道链接直接注册的用户 | `User.inviterId = channel.userId` |
| 注册人数 | 一级下线总数 | `COUNT(下线用户)` |
| 首冲人数 | 一级下线中，至少有 1 笔成功充值的用户数 | `COUNT(User.hasRecharged = true)`（该字段在充值回调成功时/后台补单时设为 true） |
| 首冲率 | 注册转首冲的转化率（冷启动核心指标） | `首冲人数 / 注册人数 * 100%` |
| 老客充值人数 | 一级下线中，成功充值次数 >= 2 的用户数 | 充值2次及以上的独立用户数 |
| 充值总额（毛额） | 一级下线所有成功充值的 `actualAmount` 合计 | `SUM(actualAmount)` |
| 通道手续费 | 每笔按该笔所用支付通道的 `payFeeRate` 百分比扣除 | `SUM(actualAmount * payFeeRate / 100)` |
| 净充值额 | 毛额 - 手续费 | `SUM(actualAmount * (1 - payFeeRate / 100))` |
| 人均充值 | 充值总额 / 首冲人数 | 衡量渠道用户质量 |

## 三、数据库设计

### 3.1 新增 MarketingChannel 模型

文件：`packages/database/prisma/schema.prisma`，添加在文件末尾 `DailyStats` 模型之后：

```prisma
// ================================
// 渠道链接相关表
// ================================

/**
 * 渠道链接表
 * @description 管理员创建的推广渠道，绑定某个用户的邀请码用于追踪推广效果
 * 每个渠道绑定唯一用户，避免统计数据重叠
 */
model MarketingChannel {
  id          Int       @id @default(autoincrement())
  name        String    @db.VarChar(100)               // 渠道名称（如"Facebook推广"）
  userId      Int       @unique                        // 关联用户ID（unique：一个用户只能绑定一个渠道）
  remark      String?   @db.VarChar(500)               // 备注说明
  isActive    Boolean   @default(true)                 // 是否启用

  createdAt   DateTime  @default(now())
  updatedAt   DateTime  @updatedAt

  // === 关联关系 ===
  user        User      @relation(fields: [userId], references: [id])

  // === 索引 ===
  @@index([isActive], map: "idx_marketing_channel_active")
  @@map("marketing_channels")
}
```

### 3.2 User 模型添加反向关联

在 `User` 模型的 `// === 关联关系 ===` 区域（约第83行之后），添加：

```prisma
  // 渠道链接关联
  marketingChannel  MarketingChannel?
```

**位置参考：** 放在 `onlineStatus UserOnlineStatus?` 之后、`@@index` 之前。

### 3.3 关键设计决策

| 决策 | 原因 |
|------|------|
| `userId` 加 `@unique` | 确保每个渠道绑定不同用户，避免多渠道共享同一用户导致统计数据重叠 |
| 不在 User 表加 `channelId` | 渠道是纯管理层概念，不入侵用户业务模型 |
| 不修改注册流程 | 渠道追踪完全基于已有的邀请关系，零侵入 |
| 首冲使用 `hasRecharged` 字段 | 避免关联子查询，该字段在充值回调和后台补单时均会设为 true，数据一致 |

### 4.4 邀请链接生成

```typescript
import { getConfig } from '@/lib/config';

/**
 * 生成渠道邀请链接
 * @description 域名从 GlobalConfig 的 site_domain 键获取，禁止硬编码
 */
async function buildInviteLink(inviteCode: string): Promise<string> {
  const domain = await getConfig<string>('site_domain', '');
  const normalizedDomain = domain.startsWith('http') ? domain : `https://${domain}`;
  const cleanDomain = normalizedDomain.replace(/\/$/, '');
  return `${cleanDomain}/register?ref=${inviteCode}`;
}
```

### 4.5 批量统计计算逻辑（列表页核心）

**设计原则：无论当前页有多少个渠道，查询次数固定为 5 次，不随渠道数量增长。**

```typescript
/**
 * 批量计算多个渠道的统计数据
 * @param userIds - 当前页所有渠道关联的 userId 数组
 */
async function batchCalculateStats(userIds: number[]): Promise<Map<number, ChannelStats>> {
  if (userIds.length === 0) return new Map();

  // === 查询1: 注册人数（按 inviterId 分组 COUNT）===
  const registerCounts = await prisma.user.groupBy({
    by: ['inviterId'],
    where: { inviterId: { in: userIds } },
    _count: { id: true },
  });

  // === 查询2: 首冲人数（下线中 hasRecharged = true 的用户数）===
  // 注意：hasRecharged 在充值回调成功时和后台补单时被设为 true
  const firstRechargeCounts = await prisma.user.groupBy({
    by: ['inviterId'],
    where: {
      inviterId: { in: userIds },
      hasRecharged: true,
    },
    _count: { id: true },
  });

  // === 查询3: 每个下线用户的充值订单数（用于计算老客）===
  const rechargeByUser = await prisma.rechargeOrder.groupBy({
    by: ['userId'],
    where: {
      user: { inviterId: { in: userIds } },
      status: 'PAID',
      actualAmount: { not: null },
    },
    _count: { id: true },
  });

  // === 查询4: 下线用户→所属渠道的映射 ===
  const downlineUserIds = rechargeByUser.map(r => r.userId);
  let userToInviter = new Map<number, number>();
  if (downlineUserIds.length > 0) {
    const userInviters = await prisma.user.findMany({
      where: { id: { in: downlineUserIds } },
      select: { id: true, inviterId: true },
    });
    userToInviter = new Map(
      userInviters
        .filter(u => u.inviterId !== null)
        .map(u => [u.id, u.inviterId!])
    );
  }

  // === 查询5: 所有相关充值订单（用于金额和手续费计算）===
  const ordersForFee = await prisma.rechargeOrder.findMany({
    where: {
      user: { inviterId: { in: userIds } },
      status: 'PAID',
      actualAmount: { not: null },
    },
    select: {
      userId: true,
      actualAmount: true,
      channel: { select: { payFeeRate: true } },
    },
  });

  // === 汇总计算 ===
  const statsMap = new Map<number, ChannelStats>();

  // 初始化所有渠道的统计为 0
  for (const uid of userIds) {
    statsMap.set(uid, {
      registerCount: 0,
      firstRechargeCount: 0,
      firstRechargeRate: '0.00',
      repeatRechargeCount: 0,
      totalGrossRecharge: '0.00',
      totalFee: '0.00',
      totalNetRecharge: '0.00',
      avgRechargePerUser: '0.00',
    });
  }

  // 填充注册人数
  for (const r of registerCounts) {
    const stats = statsMap.get(r.inviterId!);
    if (stats) stats.registerCount = r._count.id;
  }

  // 填充首冲人数 + 计算首冲率
  for (const r of firstRechargeCounts) {
    const stats = statsMap.get(r.inviterId!);
    if (stats) {
      stats.firstRechargeCount = r._count.id;
      stats.firstRechargeRate = stats.registerCount > 0
        ? (r._count.id / stats.registerCount * 100).toFixed(2)
        : '0.00';
    }
  }

  // 填充老客充值人数
  const repeatCountByInviter = new Map<number, number>();
  for (const r of rechargeByUser) {
    const inviterId = userToInviter.get(r.userId);
    if (inviterId !== undefined && r._count.id >= 2) {
      repeatCountByInviter.set(inviterId, (repeatCountByInviter.get(inviterId) ?? 0) + 1);
    }
  }
  for (const [inviterId, count] of repeatCountByInviter) {
    const stats = statsMap.get(inviterId);
    if (stats) stats.repeatRechargeCount = count;
  }

  // 填充金额统计（逐笔计算手续费，按渠道归类）
  const grossByInviter = new Map<number, number>();
  const feeByInviter = new Map<number, number>();
  for (const order of ordersForFee) {
    const inviterId = userToInviter.get(order.userId);
    if (inviterId === undefined) continue;

    const amount = Number(order.actualAmount);
    const feeRate = Number(order.channel?.payFeeRate ?? 0);
    const fee = amount * feeRate / 100;

    grossByInviter.set(inviterId, (grossByInviter.get(inviterId) ?? 0) + amount);
    feeByInviter.set(inviterId, (feeByInviter.get(inviterId) ?? 0) + fee);
  }

  for (const uid of userIds) {
    const stats = statsMap.get(uid)!;
    const gross = grossByInviter.get(uid) ?? 0;
    const fee = feeByInviter.get(uid) ?? 0;
    stats.totalGrossRecharge = gross.toFixed(2);
    stats.totalFee = fee.toFixed(2);
    stats.totalNetRecharge = (gross - fee).toFixed(2);
    stats.avgRechargePerUser = stats.firstRechargeCount > 0
      ? (gross / stats.firstRechargeCount).toFixed(2)
      : '0.00';
  }

  return statsMap;
}
```

## 六、边界情况与防御

| 场景 | 处理方式 |
|------|----------|
| 创建渠道时 userId 已被其他渠道绑定 | Prisma 抛出 P2002 错误 → 捕获并返回 `CHANNEL_USER_EXISTS: 该用户已被其他渠道绑定` |
| 创建渠道时 userId 对应的用户不存在 | 服务层查询用户并校验 → 返回 `USER_NOT_FOUND` |
| 创建渠道时用户状态为 BANNED | 服务层校验状态 → 返回 `USER_BANNED: 该用户已被封禁` |
| 渠道绑定的用户后来被封禁 | 不影响渠道记录和统计，但链接注册的下线不会获得返佣（现有逻辑已处理） |
| 渠道绑定的用户没有任何下线 | 所有统计均为 0，页面正常显示 |
| 渠道被删除后再创建绑定同一用户 | 允许，因为旧渠道记录已删除，unique 约束不冲突 |
| 同时有两个管理员尝试绑定同一用户 | 数据库 unique 约束保证只有一个成功，另一个返回友好错误 |
| 充值订单的 `actualAmount` 为 null | 统计查询中已添加 `actualAmount: { not: null }` 过滤 |
| 充值使用的支付通道被删除 | `channel` 关联查询返回 null → `payFeeRate` 默认为 0（手续费为 0，保守处理） |
| 列表页无渠道数据 | ProTable 显示空状态，无需特殊处理 |

## 七、技术约束

| 约束 | 说明 |
|------|------|
| 零侵入业务逻辑 | 不修改现有注册/邀请/返佣流程 |
| 链接格式一致 | `https://{site_domain}/register?ref={inviteCode}`，域名从 `getConfig('site_domain', '')` 获取 |
| userId unique | 一个用户只能绑定一个渠道 |
| 不允许改绑用户 | PUT 接口的 Zod schema 不包含 userId 字段 |
| 批量统计 | 列表页固定 5 次查询，不随渠道数增长 |
| 操作日志 | 创建/更新/删除均记录 `AdminOperationLog` |
| 金额精度 | 所有金额 `toFixed(2)` |
| 中文界面 | 后台所有文案用中文 |
| Remix Icon | 禁止 Emoji |
| 金额格式化 | `formatCurrency`（从 `@/utils/format` 导入） |
| 时间组件 | `TimeDisplay`（从 `@/components/common` 导入） |
| 删除请求 | 使用 `del`（不是 `delete`，从 `@/utils/request` 导入） |
| 权限控制 | 所有路由使用 `withAdminAuth`（从 `@/middleware/auth` 导入） |
